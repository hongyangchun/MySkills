# Doto 从笔记到写作

> Writing is bigger than writing. —— Bob Doto
> ZK 引导写作但不替代写作。它帮你找到方向，但笔在你手里。

## 核心原则

1. **ZK 不会告诉你写什么**：笔记集群只能指出"你在这个话题上积累了足够多的想法"，选题最终是你的决定
2. **重写，不要复制**：把笔记内容复制粘贴凑成文章 = 最差劲的写作。必须根据论证流重写
3. **先短后长**：短内容测试想法，获得反馈后再投入更长篇幅
4. **记录"下次从哪继续"**：每次写作会话结束前，必须写一句下一步开始建议

---

## 渐进式发布管道

```
主笔记集群 → 短内容 → 博客 → 长文章 → 书籍
     ↑                              │
     └──── 反馈 + 新想法 ←──────────┘
```

| 形式 | 字数量级 | 来源 | 目的 |
|------|---------|------|------|
| **短内容**（推文/朋友圈） | 50-300 字 | 单条 Zettel | 快速测试想法是否引起共鸣 |
| **博客** | 800-2000 字 | 一个主题的 Zettel 集群 | 围绕主题的连贯论述 |
| **长文章** | 3000-8000 字 | 多个 Structure Note | 更严谨的多论点论证 |
| **书籍** | 50000+ 字 | 多篇长文章整合 | 系统性输出 |

### 升级决策

"现在该写短内容还是博客？"的判断标准：

- 这个话题只有 1-3 条 Zettel → 写短内容，投石问路
- 这个话题有 5+ 条 Zettel + 一个 Structure Note → 写博客
- 这个话题有多个 Structure Note → 考虑长文章
- 发布了 3+ 篇相关长文章 → 考虑写书

## 从笔记到草稿的 8 步工作流

### Step 1：选题 — 找成熟的笔记集群

1. 使用平台的 search 操作搜索潜在话题
2. 寻找连接密度高的区域：一个话题下 Zettels 多 + 彼此连接多 = 成熟
3. **Folgezettel 成熟度信号**：同一个主题号下有深长的 ID 链（如 `4.3a1b`、`4.6c2a`）→ 这条线值得写

> "找到 3-5 条相关的 Zettels"是起步的最低条件

### Step 2：创建 Structure Note

用平台的 create 操作创建 Structure Note：

```
---
objectType: Structure Notes
---

# {文章标题} — 结构笔记

## 核心论点
{一句话总结你想论证什么}

## 论点 1：{子标题}
- @{Zettel标题}({ID}) — {这条笔记支持论点的什么}
- @{Zettel标题}({ID}) — {补充视角}

## 论点 2：{子标题}
- @{Zettel标题}({ID})

## 论证缺口
- {缺案例的地方}
- {缺反驳的环节}

## 写作顺序建议
1. 先写论点 2（因为材料最充分）
2. 再写论点 1
3. 补充过渡
```

### Step 3：审查 — 检查逻辑和素材

对照 Structure Note 检查：
- **逻辑断裂**：论点之间有跳跃吗？
- **论据缺失**：缺案例/数据/引用吗？
- **论点重复**：两个论点说的是同一件事吗？

### Step 4：补充 — 回到笔记创建

审查发现的缺口，回到 `doto-create` 补充对应的笔记。如果缺口是事实性的（需要查资料），先记 Reference Note。

### Step 5：起草 — 重写，不是复制

**这是最关键的一步。**

- ❌ 不要：把 Zettel 的内容一段段复制粘贴到草稿里
- ✅ 要做：把草稿当成全新文字来写，在需要论据时引用笔记中的核心观点和案例
- Zettel 是你的"原料仓库"，不是"预制件"。你需要把它们**消化后重新烹饪**

Doto："Your Zettelkasten will show you what's possible, but it won't write for you."

### Step 6：编辑 — 删减、重组、过渡

- 删掉不服务于核心论点的内容
- 调整段落顺序，让论证流顺畅
- 在段落间添加过渡句
- 大声朗读，感受节奏

### Step 7：发布 — 选择合适的平台

| 内容形式 | 建议平台 |
|---------|---------|
| 短内容 | Twitter/X / 朋友圈 / 即刻 |
| 博客 | 个人网站 / 公众号 / Medium / Substack |
| 长文章 | 个人网站 / 公众号 / 少数派 |
| 书籍 | 出版社投稿 / 自出版 |

### Step 8：记录使用 — 更新 Zettel 的 used_in

写作完成后，回到每条被引用的 Zettel，在正文末尾添加：

```
📝 已用于：{文章标题} ({发布日期})
```

或使用平台的 append 操作追加。这能让 Doto 的"使用记录"要素落地，未来回顾时能快速找到"这个想法在哪篇文章里用过了"。

## 写作管理系统

整合 Doto 书中第 9 章的三种管理工具。

### 每日日记

使用平台的 quickCapture 或 Daily Note 功能：

```
## 今日收件箱
- 💡 {临时想法1}
- 💡 {临时想法2}

## 今日写作
- {写了什么/改了什么}

## 今日连接
- {建立了哪些有趣的笔记连接}

## 明日建议
- {下次从哪继续？一句话}
```

### CLOG（创意日志）— 每次写作会话后

使用平台的 create 操作创建 CLOG 条目（或在 Obsidian 中直接创建一个 `.md` 文件，或 Capacities 中创建 CLOG 对象）：

```
---
objectType: CLOG
project: {项目名}
---

## 本次完成
- {具体做了什么}

## 待处理
- {接下来要做的}

## 心得 / 发现
- {写作过程中的发现}

## 下次开始建议
- {一句话，下次打开就知道从哪继续}
```

CLOG 的核心价值是什么：
- 这是 Doto 书中**最被低估的工具**
- 它的核心是一句："下次从哪继续"。不要让你明天打开文件时一片空白
- 哪怕只写了一句话，"打开 3.2a 那条笔记，看看能不能发展出论证 B"——这也是有价值的

### 任务驱动项目管理（有明确截止日时）

当你有明确的写作截止日（如"下周要发一篇博客"）：

1. **定义项目**：一篇博客 / 一篇文章 / 一本书
2. **头脑风暴任务**：列出所有需要做的事
3. **收集资源**：从 ZK 中搜索相关笔记 → 创建 Structure Note
4. **只做今天的任务**：不要提前焦虑明天的
5. **记录进展**：写入 CLOG
6. **反思调整**：任务完成/未完成都记为什么

## 写作哲学

> "Writing is bigger than writing." —— Bob Doto

写作不是编辑文字。写作包含：

捕捉想法 → 提炼笔记 → 连接关联 → 组织论点 → 起草 → 编辑 → 发布 → 获取反馈 → 回到捕捉

**ZK 帮你完成写作的前半段（捕捉→连接→组织），但后半段（起草→编辑→发布）的每一个字都需要你亲自写。**

---

## 章末检查清单

每次写作会话结束时检查：

- [ ] 是否找到了足够成熟的笔记集群（≥ 3 条）？
- [ ] 是否创建了 Structure Note 作为写作大纲？
- [ ] 草稿是否**重写**了笔记内容（而非复制粘贴）？
- [ ] 发布后是否更新了 Zettel 的 used_in？
- [ ] 是否在 CLOG / Daily Note 记录了"下次从哪继续"？
- [ ] 如果有截止日，是否只关注了今天的任务？

## 常见问题

**Q: Structure Note 和草稿有什么区别？**
A: Structure Note = 蓝图（Zettel 链接 + 论点排列），草稿 = 建筑（连贯的文字）。Structure Note 是你的施工图纸，不是成品。

**Q: 什么时候该放弃一个写作项目？**
A: 如果 Structure Note 里"论证缺口"那一栏迟迟填不上，且你没有动力回到这个项目 → 搁置，不是放弃。ZK 里的笔记不会过期，三年后你积累够了自然会回来。

**Q: 我该不该追求日更？**
A: Doto 的方法论不追求频率，追求质量。一条被反复连接、被 5 篇文章引用的 Zettel 比 100 条孤岛笔记有价值。写作同理——一篇扎实的博客比十篇水文有意义。
