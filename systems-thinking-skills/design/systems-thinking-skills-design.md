## 1. 概述

### 1.1 目标

将《系统之美》的核心框架转化为一套可交互的引导式诊断 skill，帮助用户：

- **分析系统结构**——用存量/流量、反馈回路、延迟等工具拆解复杂问题
- **诊断系统顽疾**——识别 8 大系统基模，找到"为什么越努力越糟"的根因
- **寻找变革杠杆**——用 12 个杠杆点框架找到高杠杆干预机会

### 1.2 设计原则

1. **引导式诊断**——不是提供概念词典，而是通过提问引导用户自己发现系统结构
2. **完整自包含**——每个 skill 自带足够的解释、案例和检查清单，不依赖用户预先读过书
3. **场景驱动触发**——用真实的用户场景描述作为触发词，而非生硬的术语
4. **模块化**——入口 skill 做路由，子 skill 各自聚焦，独立可触发

### 1.3 与现有 `complex-problem-solver` 的关系

当前已有 `complex-problem-solver` skill 将系统思维作为 12 大思维模型之一。本套件是**深度专精版**——不追求覆盖面，而是把系统思维这一条线做深做透，提供完整的诊断流程和实操框架。两者互补，不冲突。

```
├── systems-thinking-archetypes  (基模诊断：顽疾识别)

```

### 2.2 文件结构

```

├── systems-thinking/
│   └── SKILL.md          # 主入口：路由分发
├── systems-thinking-core/
│   └── SKILL.md          # 系统基础
├── systems-thinking-archetypes/
│   └── SKILL.md          # 基模诊断
└── systems-thinking-leverage/
    └── SKILL.md          # 杠杆点分析
```

### 2.3 路由逻辑

主入口 skill 根据用户意图分三类场景路由：

| 意图信号 | 路由目标 | 典型用户表达 |
|----------|----------|-------------|
| 理解结构 | core | "这个系统的结构是什么""帮我理清反馈回路""为什么会有这种趋势" |
| 诊断顽疾 | archetypes | "这个问题反复出现""越努力越糟""投入了但没效果""感觉陷入恶性循环" |
| 寻求变革 | leverage | "怎么改变这个局面""从哪里入手""怎么做才能真正生效" |
| 模糊/学习 | 提问澄清 | "我想学系统思维""什么是反馈回路" → 先问再路由 |

**路由机制**：主入口 SKILL.md 中包含意图判断指南，assistant 读取后自行判断并调用对应子 skill 的 Skill tool 加载。

## 3. 各 Skill 详细设计

### 3.1 `systems-thinking`（主入口）

**Frontmatter**：
```yaml
name: systems-thinking
description: >-
  系统思维工具箱主入口。当用户需要分析复杂问题的结构、诊断反复出现的顽疾、
  或寻找变革切入点时使用。基于 Donella Meadows《系统之美》。
  触发词：系统思维、系统思考、system thinking、反馈回路、杠杆点、
  这个问题反复出现、越努力越糟、系统陷阱、系统结构。
```

**核心内容**（约 800 字）：

1. **意图识别指南**——三类场景的关键词和语义特征
2. **路由表**——每种意图对应加载哪个子 skill
3. **澄清话术**——当意图不明确时如何提问
4. **概念速查**——如果用户只是想快速了解某个概念，主入口可直接简答，不必每次都路由

**交互模式**：
- 用户表达需求 → 主入口判断意图 → 如果清晰，直接加载子 skill 并说明"我将用 XX 模块来帮你分析"
- 如果模糊，问一个问题："你是想理解这个系统的结构、诊断为什么问题反复出现、还是想找到改变现状的切入点？"

### 3.2 `systems-thinking-core`（系统基础）

**Frontmatter**：
```yaml
name: systems-thinking-core
description: >-
  系统思维基础模块：用存量/流量、反馈回路、延迟等工具分析系统结构。
  当用户需要理解一个系统是如何运作的、为什么会产生某种行为模式时使用。
  通常由 systems-thinking 主入口路由触发，也可独立使用。
```

**核心内容**（约 3500 字）：

#### 概念模块

**A. 系统的三要件**
- 元素（Elements）：系统中看得见的部分
- 相互连接（Interconnections）：元素之间的关系和规则
- 目标/功能（Purpose/Function）：系统实际运作的方向（不一定是声称的目标）
- 关键洞察：改变元素最容易但效果最小，改变相互连接和目标才是高杠杆

**B. 存量与流量（Stocks & Flows）**
- 存量：系统中某时刻的积累量（如银行余额、团队人数、水库水量）
- 流量：单位时间内的流入/流出（如收入/支出、招聘/离职、注水/排水）
- 关键洞察：存量变化慢，起到缓冲和延迟作用；存量是系统的"记忆"
- 诊断问题："这个系统里什么是积累的？什么是流入流出的？"

**C. 反馈回路（Feedback Loops）**
- 增强回路（Reinforcing）：自我强化，导致指数增长或崩溃
  - 识别特征："越多越多"或"越少越少"
  - 案例：复利、口碑传播、军备竞赛、恶性通胀
- 调节回路（Balancing）：自我修正，趋向目标值
  - 识别特征："趋向某个水平""会被拉回来"
  - 案例：恒温器、体温调节、供需价格
- 关键洞察：一个系统中通常有多个回路同时作用，系统行为取决于哪个回路占主导
- 诊断问题："什么在推动增长？什么在拉回？"

**D. 延迟（Delays）**
- 延迟让系统反应滞后，导致振荡和过冲
- 物理延迟 vs 感知延迟 vs 反应延迟
- 关键洞察：延迟越长，系统越容易过冲和振荡
- 诊断问题："从行动到效果之间有多长的延迟？"

**E. 系统行为模式**
- 指数增长（增强回路主导）
- S 型增长（增强回路 + 调节回路）
- 振荡（调节回路 + 延迟）
- 过冲与崩溃（增强回路 + 调节回路 + 长延迟 + 不可逆阈值）

#### 诊断流程（引导式，逐步推进）

```
Step 1: 界定系统
  ── "你要分析的是什么系统？它的边界在哪里？"
  ── 输出：系统名称 + 核心元素 + 目标/功能

Step 2: 识别存量与流量
  ── "这个系统里，什么是逐渐积累的？什么是流入和流出的？"
  ── 输出：关键存量 + 流入/流出清单

Step 3: 绘制反馈回路
  ── "什么在推动它增长/扩大？（增强回路）"
  ── "什么在限制它、拉回它？（调节回路）"
  ── 输出：回路图 + 回路之间的相互作用

Step 4: 标注延迟
  ── "从你采取行动到看到效果，中间隔了多久？"
  ── 输出：延迟位置 + 延迟时长

Step 5: 预测行为模式
  ── "基于以上结构，这个系统最可能表现出什么行为？"
  ── 输出：行为模式判断 + 对用户当前困惑的解释
```

#### 交互要点
- 每个 Step 只问 1-2 个问题，等用户回答后再推进
- 用户回答后，用系统思维语言重新组织（"你描述的其实是一个增强回路..."）
- 适当用案例类比帮助理解（"这就像恒温器..."）
- 结束时输出一张"系统结构摘要"：存量 → 流量 → 回路 → 延迟 → 行为模式

### 3.3 `systems-thinking-archetypes`（基模诊断）

**Frontmatter**：
```yaml
name: systems-thinking-archetypes
description: >-
  系统基模诊断模块：识别 8 大系统陷阱（政策阻力、公地悲剧、竞争升级、
  成功者通吃、转嫁负担、目标侵蚀、规则规避、目标错位），
  并给出针对性破解策略。当用户描述的问题具有"反复出现""越努力越糟"
  "短期有效长期反弹"等特征时使用。
```

**核心内容**（约 4000 字）：

#### 8 大基模详解

每个基模包含：症状描述 | 底层结构 | 破解杠杆点 | 真实案例

**1. 政策阻力（Policy Resistance）**
- 症状：投入了资源，短期见效，但很快反弹回原状
- 结构：系统内多方利益相关者有不同目标，调节回路互相拉扯
- 案例：禁毒→毒品价格涨→更多人冒险贩毒；限价→供给减少→黑市
- 破解：找到各方背后的真实目标，设计让所有目标都能部分实现的方案；或彻底重新定义系统目标

**2. 公地悲剧（Tragedy of the Commons）**
- 症状：每个人都在理性地最大化自身利益，但共享资源被耗尽
- 结构：共享资源 + 个体使用不受限制 + 缺乏反馈延迟
- 案例：过度捕捞、公司共享预算被各部门争抢
- 破解：让使用者直接承担资源耗尽的成本；或引入产权/配额机制

**3. 竞争升级（Escalation）**
- 症状：双方不断加码，都停不下来，最终双输
- 结构：两个增强回路互相 feeding，A 的增长刺激 B 增长，反之亦然
- 案例：军备竞赛、价格战、广告战
- 破解：单方面退出（需要勇气和多轮博弈信任）；改变规则让升级不再有回报

**4. 成功者通吃（Success to the Successful）**
- 症状：赢家持续获得更多资源，输者越来越弱
- 结构：增强回路将资源从输者转移到赢者
- 案例：富人越富、明星效应、大公司垄断
- 破解：多元化评价标准；定期重置；对赢家设上限

**5. 转嫁负担（Shifting the Burden to the Intervener）**
- 症状：依赖外部干预解决问题，自身能力逐渐萎缩
- 结构：短期症状缓解（调节回路）削弱了根本解决方案（另一个调节回路）
- 案例：依赖止痛药不治本、外援依赖、外包核心能力
- 破解：区分症状治疗和根本治疗；始终保持对根本方案的关注和投入

**6. 目标侵蚀（Drift to Low Performance）**
- 症状：标准越来越低，绩效持续下滑，但没人觉得有问题
- 结构：实际表现 → 降低期望 → 更低努力 → 更低表现（增强回路，负向）
- 案例："以前觉得不行，现在也习惯了""行业平均水平就这样"
- 破解：维持绝对标准不受当前表现影响；引入外部标杆

**7. 规则规避（Rule Beating）**
- 症状：人们遵守规则的字面意思但违背了规则的精神
- 结构：规则作为调节回路，但被规避后系统目标被架空
- 案例：KPI 至上导致数据造假、教学为了考试
- 破解：重新设计规则使其与系统真实目标一致；减少刚性规则，增加判断空间

**8. 目标错位（Seeking the Wrong Goal）**
- 症状：越努力越偏离真正的目标
- 结构：系统目标设定错误，优化了错误的指标
- 案例：GDP 至上忽视幸福度、点击量至上忽视内容质量
- 破解：重新审视系统的真正目标；用多维指标替代单一指标

#### 诊断流程

```
Step 1: 倾听问题描述
  ── "描述一下你遇到的问题：它是什么样的？什么时候出现的？"
  ── 关注关键词：反复、反弹、恶化、停不下来、越...越...

Step 2: 症状匹配
  ── 将用户描述与 8 大基模的症状特征比对
  ── 可能匹配 1-2 个候选基模

Step 3: 确认诊断
  ── 针对候选基模提问验证
  ── "是不是投入了但很快反弹？" → 政策阻力
  ── "是不是双方都在加码停不下来？" → 竞争升级

Step 4: 解释底层结构
  ── 用反馈回路语言解释为什么会产生这个症状
  ── "你描述的情况背后，其实是一个增强回路在..."

Step 5: 给出破解策略
  ── 针对确认的基模给出具体破解方向
  ── 关联到杠杆点（引导用户进入 leverage 模块做深入分析）
```

#### 交互要点
- 不要一次列出 8 个基模让用户选——先听描述，主动匹配
- 确认时要给出基模的"典型症状"让用户对号入座
- 破解策略要具体可操作，不停留在"改变系统目标"这种抽象层面
- 鼓励用户进入 leverage 模块做更深入的变革分析

### 3.4 `systems-thinking-leverage`（杠杆点分析）

**Frontmatter**：
```yaml
name: systems-thinking-leverage
description: >-
  系统杠杆点分析模块：用 Meadows 的 12 个干预杠杆点框架，
  从最无效到最强大逐一扫描，帮助用户找到高杠杆变革机会。
  当用户想要改变一个系统、寻找入手点时使用。
```

**核心内容**（约 3500 字）：

#### 12 个杠杆点（从低到高）

完整对应 Meadows 原文《Places to Intervene in a System》的 12 个干预点，从最无效（#12）到最强大（#1）。每个包含：定义 | 为什么有效/无效 | 干预示例。

**低杠杆区（#12–#9）——有效但有限**

**#12 参数与数字（Numbers）**
- 调整常量和参数：税率、预算、配额、阈值、标准
- 为什么低效：只调数字不改变结构，效果有限且容易被绕过
- 示例：调高罚款金额（人们只是更小心地违规）、调整最低工资

**#11 物质存量与流量（Material Stocks and Flows）**
- 改变系统的物理结构：建水库、修路、增加产能、改变节点间的物质连接
- 为什么低效：耗时漫长、成本高昂，且不改变系统的运行逻辑
- 示例：修建蓄水池增加蓄水能力、扩建生产线

**#10 缓冲器（Buffers）**
- 调整稳定存量的相对大小：库存、储备金、人才池
- 为什么偏低：缓冲器越大系统越稳定，但过大则迟钝；改变缓冲器大小见效慢
- 示例：增加安全库存、设立应急基金、培养人才储备
- 与 #11 的区别：#11 是改变物理结构本身，#10 是调整已有存量相对于流量的比例大小

**#9 延迟（Delays）**
- 调整响应时间的长短：缩短信息延迟、改变决策延迟
- 为什么偏低但关键：延迟太短系统过度反应，延迟太长系统振荡过冲
- 示例：加快市场反馈数据传递、缩短审批流程、放慢退出决策以避免恐慌
- 关键洞察：不是越短越好——关键延迟要与系统变化速率匹配

**中杠杆区（#8–#5）——结构性改变**

**#8 负反馈回路（Negative Feedback Loops）**
- 加强自我修正机制：让反馈更灵敏、更准确、更有力
- 为什么中效：改善系统稳定性，但不改变系统的目标和方向
- 示例：增加质量检查频率、建立预警机制、强化制衡机制
- 关键：反馈回路的强度要与它要修正的冲击力匹配

**#7 正反馈回路（Positive Feedback Loops）**
- 调整增长引擎的增益：加速好的增强回路、刹车坏的增强回路
- 为什么中效：增强回路是指数增长/崩溃的引擎，调节增益影响巨大
- 示例：建立推荐奖励机制（加速好的）、限制金融杠杆率（刹车坏的）
- 关键：减小坏回路的增益比增大好回路的增益通常更有效

**#6 物质流路径与边界（Material Flow Pathways and Crossing of Boundaries）**
- 改变谁与谁连接：重组物质流动路径、重新划定系统边界
- 为什么较高：改变了系统的拓扑结构，即"谁连接到谁"
- 示例：改变供应链上下游关系、将决策权从总部下放到区域、重新设计部门协作流程
- 关键：不是信息流而是物质流——谁拿到资源、谁承担责任

**#5 规则（Rules）**
- 改变游戏规则：准入条件、奖惩机制、权力分配、约束条件
- 为什么较高：规则定义了系统的激励结构，决定了参与者的行为
- 示例：改变选举规则、调整绩效考核方式、修改准入门槛
- 关键：规则改变后，系统内所有人的行为都会跟着变

**高杠杆区（#4–#1）——根本性变革**

**#4 自组织（Self-Organization）**
- 让系统能自我进化：鼓励实验、允许多样性、降低变异成本
- 为什么很高：让系统获得学习和适应能力，能自行发现更好的结构
- 示例：内部创业机制、开源社区、生物多样性保护
- 关键：自组织需要"变异 + 选择 + 保留"三个条件同时存在

**#3 系统目标（Goals）**
- 改变系统追求的目标：从增长转向可持续、从效率转向韧性
- 为什么极高：目标改变，所有下游结构——规则、回路、参数——都会跟着重新排列
- 示例：从"GDP 增长"转向"国民幸福"、从"考试分数"转向"能力培养"
- 关键：区分系统的声称目标和实际目标（看行为而非口号）

**#2 范式（Paradigms）**
- 改变看待世界的基本假设：从机械论到生态论、从线性到系统
- 为什么最高之一：范式决定了目标、规则、信息流——一切下游都源于范式
- 示例：从"疾病治疗"到"健康管理"、从"征服自然"到"与自然共生"
- 改变范式的方法：持续暴露旧范式的矛盾，同时构建新范式的成功案例

**#1 超越范式（Transcending Paradigms）**
- 意识到没有哪个范式是终极真理，保持灵活和谦逊
- 为什么是最高：不执着于任何单一视角，可以自由切换范式
- Meadows 的终极建议：保持开放，不要被任何一种思维框架困住

> 编号说明：#12（最低杠杆）→ #1（最高杠杆），与 Meadows 原文一致。诊断时按 #12→#1 从低到高扫描。

#### 诊断流程

```
Step 1: 理解系统和目标
  ── "你想改变的是什么系统？它当前的目标是什么？"
  ── "你希望它变成什么样？"

Step 2: 低杠杆扫描（#12–#9）
  ── #12 参数："有没有数字可以调？效果会怎样？"
  ── #11 物质结构："需要改变物理基础设施吗？"
  ── #10 缓冲器："关键存量相对于流量够不够大？太大还是太小？"
  ── #9 延迟："关键延迟太长还是太短？能调整吗？"
  ── 标注：哪些有干预空间，但效果有限

Step 3: 中杠杆扫描（#8–#5）
  ── #8 负反馈回路："自我修正机制够不够灵敏、有力？"
  ── #7 正反馈回路："增长引擎需要加速还是刹车？"
  ── #6 物质流路径："谁和谁之间的连接需要改变？"
  ── #5 规则："游戏规则需要怎么改？"
  ── 标注：哪些是结构性改变机会

Step 4: 高杠杆扫描（#4–#1）
  ── #4 自组织："系统有没有自我进化的能力？"
  ── #3 目标："系统的目标需要重新定义吗？"
  ── #2 范式："看待这个问题的方式需要根本性改变吗？"
  ── #1 超越范式："是否需要跳出当前框架？"
  ── 标注：最高杠杆但也最难实现的干预点

Step 5: 选择与预判
  ── 综合扫描结果，推荐 1-2 个"高杠杆且可行"的干预点
  ── 预判："如果你在这里干预，系统会怎么反应？"
  ── 提醒延迟和副作用
```

#### 交互要点
- 不要一次性列完 12 个杠杆点——按低→中→高分三段（#12-#9、#8-#5、#4-#1）推进
- 每段扫描后问"这里有没有让你眼前一亮的干预机会？"
- 推荐时区分"高杠杆但容易做"和"高杠杆但很难做"
- 务必提醒：高杠杆干预往往需要勇气和时间，效果可能延迟显现
- 结束时输出"杠杆点扫描摘要"：标注每个点的干预空间和推荐优先级

## 3.5 可视化设计

诊断过程中使用 `show_widget` 绘制 SVG 图表，帮助用户直观理解系统结构。每个模块在关键步骤自动触发可视化。

### core 模块可视化

| 诊断步骤 | 可视化类型 | 内容 |
|----------|-----------|------|
| Step 2 存量与流量 | 存量流量图 | 一个方框（存量）+ 流入箭头 + 流出箭头，标注名称和方向 |
| Step 3 反馈回路 | 因果回路图（CLD） | 节点 + 箭头，箭头标 + / - 表示同向/反向影响，回路中心标 R（增强）或 B（调节） |
| Step 5 行为模式 | 时间趋势图 | X 轴=时间，Y 轴=存量值，画出预测的趋势曲线（指数/S型/振荡/过冲） |

**因果回路图绘制规范**：
- 节点用圆角矩形，增强回路用暖色（coral/amber），调节回路用冷色（teal/blue）
- 箭头旁标注 `+`（同向变化）或 `-`（反向变化）
- 回路中心放一个大字 `R` 或 `B` 标识类型
- 如果有延迟，在对应箭头上画 `||` 标记

### archetypes 模块可视化

| 诊断步骤 | 可视化类型 | 内容 |
|----------|-----------|------|
| Step 4 解释底层结构 | 基模结构图 | 该基模的典型反馈回路结构，标注"陷阱位置"和"破解干预点" |
| Step 5 破解策略 | 对比图 | 左侧"当前陷阱结构" vs 右侧"破解后结构"，用箭头连接显示变化 |

**基模结构图示例（政策阻力）**：
```
[干预措施] --(-)--> [问题指标] --(+)--> [反弹力量] --(-)--> [问题指标]
                                        ↑ 陷阱：反弹力量抵消干预
```

### leverage 模块可视化

| 诊断步骤 | 可视化类型 | 内容 |
|----------|-----------|------|
| Step 2-4 杠杆点扫描 | 杠杆点金字塔 | 倒金字塔：底部=低杠杆（#12 参数），顶部=高杠杆（#1 超越范式），标注当前扫描到的层级 |
| Step 5 选择与预判 | 干预前后对比图 | 左侧"当前行为趋势" vs 右侧"干预后预测趋势"，标注干预位置 |

**杠杆点金字塔绘制规范**：
- 从下到上 12 层，越往上越窄（表示杠杆越高）
- 已扫描且有干预机会的层高亮（暖色填充）
- 无干预空间的层灰色淡显
- 推荐的干预点用箭头标注

### 可视化触发原则

1. **适时出现**——在用户回答关键问题后自动绘制，不是一开始就画
2. **解释先行**——先口头解释，再画图辅助，图是补充不是替代
3. **简洁为主**——每张图只表达一个核心信息，不堆砌
4. **暗色主题适配**——使用 dark mode 配色（800 填充 + 200 描边 + 100 文字）

## 4. 实现计划

### 4.1 创建顺序

1. `systems-thinking-core`（基础模块，其他模块会引用其概念）
2. `systems-thinking-archetypes`（基模诊断，引用 core 的反馈回路概念）
3. `systems-thinking-leverage`（杠杆点分析，引用前两者）
4. `systems-thinking`（主入口，最后写路由逻辑）

### 4.2 技术要点

- 存储：用户级 `~/.workbuddy/skills/` 下各自独立目录
- 每个 SKILL.md 包含 frontmatter + 完整内容，不依赖外部文件
- 主入口通过 frontmatter description 中的触发词被激活
- 子 skill 可被主入口路由调用，也可独立触发
- 所有内容使用中文（用户默认语言）

### 4.3 验证方式

创建完成后，用以下场景测试路由是否正确：
- "我们团队的问题反复出现，投入了很多但总是反弹" → 应路由到 archetypes
- "帮我分析一下我们公司这个业务系统的结构" → 应路由到 core
- "我想从根本上改变现在的局面，从哪里入手" → 应路由到 leverage
